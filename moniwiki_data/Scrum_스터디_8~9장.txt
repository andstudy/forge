= 8. 스크럼과 조직 =
 * 조직에 미치는 영향
  * 스크럼의 목표는 방해물 제거
  * 스크럼은 조직에 생산성에 방해가 되는 것이 무엇인지를 알 수 있는 기회를 제공함
   * 장애물들을 매일 지적하여 해결되게 함 (''아래로부터의 프로세스 혁신'')
   * 프로젝트 포스트 모르템 : 프로젝트 개발이 완료된 이 후에 개발 과정을 돌이켜 분석
    --> 효과가 적고, 너무 느린 방법
  * 관리자는 생산성을 최상으로 유지할 책임이 있음
   * 스크럼은 생산성을 높일 수 있는 방법이 적힌 목록을 제공
   * 장애물 발견을 통해 조직이 생산성을 어떻게 침해하는지 명확히 알 수 있는 방법을 제공함
----
 * 변화를 이끌어내는 스크럼 마스터
  * 스크럼 마스터가 장애물을 제거할 수 있는 권한을 위임 받는 것이 중요
  * 주의를 기울여 지켜보기
   * 일일 스크럼에서는 직접 언급된 장애물을 파악
   * 일상 대화에서 모두가 당연시 여기는 장애물을 찾아낼 수 있음
----
 * 장애물 예시 1
  * 선임 개발자 한명이 특정 소프트웨어 라이브러리 구매 신청을 함
  * 도착을 안해 4일 동안 프로젝트 지연
  * 일일 스크럼 회의 때마다 이 문제를 장애라고 얘기함
  * 스크럼 마스터의 처리
   * 직접 연락 후 바로 라이브러리를 구매함
   * 구매 지연 이유 파악(총무부에 인력부족) 후 처리(인원 충원 권고)
  * 의문점
   * 스크럼 마스터는 왜 문제 해결을 즉시 못하고 4일이나 걸렸을까...
   * 이 후 회사에서 총무부에 인원을 충원 했을까...
----
 * 장애물 예시 2
  * 새로운 개발자에게 1년도 더 지난 구매 목록 그대로의 장비를 제공함
  * 개발자는 관리자가 잘 선택한 것으로 생각하고 그냥 받아들임
  * 스크럼 마스터의 처리
   * 일일 스크럼을 통해 문제 확인
   * 관리팀과 싸워 장비를 최신 사양 혹은 개발자 스스로 선택 할 수 있도록 함
  * 의문점
   * 새로 구입한 장비인데... 다시 최신 사양으로 구입할지... 환불??
----
 * 장애물 예시 3
  * 중요한 전사쇼에서 시스템 작동을 두 명의 선임 엔지니어가 맡음
  * 프로그래밍이나 부품 조립, 선반 준비는 다른 팀원들이 맡음
  * 일일 스크럼에서 두 선임 엔지니어는 부품 획득의 어려움을 토로
  * 스크럼 마스터의 처리
   * 선임 엔지니어에게 핸드폰을 지급
   * 신입 엔지니어에게 부품 조달 작업을 할당
  * 의문점
   * 신입 엔지니어가 맡은 일은 누가하나??
----
 * 장애물 예시 4
  * 모든 사람이 파티션으로 둘러싸인 개인 공간에 배치됨
  * 개발자들이 논의를 위해 자리 이동이 빈번함
  * 스크럼 마스터의 처리
   * 빈번한 자리 이동 때문에 업무의 효율성이 떨어지는 것을 파악
   * 작업 공간을 재배치 하여 서로 더욱 효과적으로 작업 할 수 있게 함
----
 * 장애물 예시 5
  * 스프린트 중간에 반년마다 실시하는 평가 프로세스 작업을 함
  * 스크럼 마스터의 처리
   * 모두가 당연시 여기는 업무였지만 스크럼 마스터는 장애물로 봄
   * 경영진에 스프린트가 끝날 때까지 평가 작업을 미루도록 요청
  * 의문점
   * 스프린트가 끝나면 바로 다음 스프린트가 시작될텐데...
   * 평가 프로세스를 다음 스프린트의 백로그에 등록??
----
 * 장애물 예시 6
  * 일일 스크럼 회의에서 커피를 사기 위해 잔돈을 세는 것을 자주 목격
  * 스크럼 마스터의 처리
   * 시설 관리부장에게 커피를 무료로 제공해줄 것을 요청
   * 결국 회사에서 공짜 커피를 제공해 줌
  * 의문점
   * 예전에 공짜로 제공 했었는데 모두 불평이어서 없앤건데...


= 9. 스크럼의 가치 =
 * 자발적 헌신
  * 자신의 공약을 지키려고 헌신하는 사람들에게 그들이 필요로하는 모든 권한을 부여함
   * 팀은 스스로 선택한 작업을 어떻게 할지 결정하는 권한이 있다

 * 집중
  * 스크럼은 팀이 문제점을 설정하고 거기에 집중할 수 있는 환경을 제공함
   * 업무 도중에 다른 부서의 회의, 사내 교육, 이메일 보내기, 평가 쓰기 등

 * 개방성
  * 프로젝트에 대한 모든 내용을 투명하게 공개함
   * 제품 백로그, 일일 스크럼 회의, 스프린트 리뷰 회의 등
  * 어떻게 스프린트 목표를 달성해야 하는지는 팀이 스스로 결정할 문제
  * Sun/Solaris 솔루션 스프린트 vs IBM과의 파트너

 * 존중
  * 스크럼은 여러 사람들이 하나의 팀으로 협업할 수 있는 환경을 제공함
   * 다양한 개개인을 묶어 하나의 팀으로 만들 때 팀은 강점을 얻게 됨
  * 여러 번의 스프린트를 통해 팀은 각자의 장단점과 그걸 받아들이는 법과 어떻게 조화시키며 보충할지 배움

 * 용기
  * 팀이 다른 방식으로 움직이기 위해서는 용기가 필요함
   * 스크럼의 가치를 지원할 환경을 찾아내고자 하는 용기
   * 자신의 판단을 신뢰하고 관철시키려는 용기
   * 도움이 필요할 때 다른 팀원에게 알리고 어떻게 하면 좋을지 물어볼 수 있는 용기

----
 * 느낀점
  * 책에서 처럼 행동하고 실행하는 스크럼 마스터가 있을까...
  * 스크럼을 잘 이해하는 스크럼 마스터가 없으면 스크럼은 그림의 떡
  * 스크럼을 도입해서 프로젝트가 실패한 사례가 있을텐데... 스크럼 마스터의 문제 말고...
